## 事故实录（真实案例，2026-07-03）

**任务**：用 pi-subagent 编排一个 LLM 客户端项目（guyiichat），写完 `_plan-draft.md` 后应调用 `pi_task_create` 建立任务。

**实际发生**：
1. 模型生成的工具名串了——本应调 `pi_task_create`（pi-subagent 的任务创建工具），却调成了 `analyze_image`（图像分析工具，与本任务**零关系**）。
2. 用占位符 URL `https://example.com/x.png` 喂给它，返回 400「图片输入格式/解析错误」。
3. 模型把 400 当成"参数没给对"，反复换新占位符 URL 重试同一错误工具，**循环 6 次**。
4. 更糟的是，模型在自然语言里**一边说"这是误触发、忽略它"，一边继续调用同一个会失败的图像工具**——言行不一致。

**根因**：工具名 token 生成时串了（`pi_task_create` ↔ `analyze_image`）；失败后未执行"工具持续失败就该换策略"的基本规则，而是机械重试。

## 待合入 SKILL.md 的规则（建议措辞）

### 反模式条目（加到「## 反模式（禁止）」）

```markdown
- ❌ 工具调用失败后机械重试 —— 工具返回错误时，
  必须先诊断"是否调错了工具 / 参数是否合理"，而不是原样或微调重试。
  特别警惕：① 生成错误工具名（如把 pi_task_create 串成 analyze_image
  这类任务里根本不存在的工具）；② 用占位符（example.com / "test"）
  喂工具。任何工具连续失败 2 次即停止、回到上层重新规划，
  绝不在自然语言里"一边说要停一边继续重试"。
```

### 上层自检规则（建议加到「## 重要行为说明」或独立小节）

```markdown
**工具名生成后自检（防串名事故）**：
调用工具前，确认该工具名出现在"本任务相关工具清单"里。判定标准：
- 任务是否包含该工具所属领域？例：构建 LLM 客户端 / 排版课件 / 写文档
  等任务**不含视觉/图像分析需求**，则 `analyze_image` / `mcp__4_5v_mcp__*`
  等图像工具绝不应被调用。
- 若发现调用的工具与本任务无关（领域不匹配 / 用了占位符参数），
  视为 token 串名事故，**立即停止**，重新核对工具清单后再调。
- 一次错误调用就停，不要重试。
```

### 行为约束补强（可并入现有"重要行为说明"）

```markdown
**失败即停原则**：
- 任何工具调用返回 error（HTTP 4xx / 业务错误码 / timeout），
  第一反应是"诊断"而非"重试"：检查①工具名对不对 ②参数是否真实合理（非占位符）
  ③是否在正确的上下文。
- 连续 2 次同类失败 → 强制回到上层规划，重新拆解，禁止第 3 次原样重试。
- 自然语言与工具调用必须一致：若已口头承认"这是误操作"，则后续禁止
  再发起同一错误调用。
```

## 为什么这样写有效（给维护者的设计说明）

1. **写"触发条件"而非"道德说教"**：模型识别不了"要小心"，但能识别
   "工具连续失败 2 次""生成了任务里不存在的工具名""用占位符 URL"这些
   **具体信号**。规则全部锚定在可观测的信号上。

2. **给替代动作**：只说"禁止"不够，规则里明确写了失败时该做什么——
   "停下来、核对工具名、回到上层重新规划"。

3. **带真实案例**：事故实录比抽象规则更有锚定力，能让模型在类似情境下
   回想起"上次这里翻车了"。这也是 progressive disclosure 里 references
   的正确用法——SKILL.md 放精简规则，案例放本文件按需读。

4. **聚焦本 skill 的边界**：本文档只针对"工具调用循环"这一类事故，
   不泛化成通用安全规则，避免污染 SKILL.md 的核心编排职责。
